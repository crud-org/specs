This Protocol Specification maps the [CRUD API Base Specification](/0.1/base/) to the **RESTful HTTP** transport protocol, as defined in [Protocols](/0.1/protocols/). It follows the [HTTP Protocol](https://datatracker.ietf.org/doc/html/rfc9110) and aligns with [REST architectural constraints](https://restfulapi.net/).

The five protocol concepts are realized as follows:

| Concept | HTTP realization |
|---|---|
| **Operations** | HTTP methods (`GET`, `POST`, `PATCH`, `PUT`, `DELETE`) combined with URL patterns (`/:collection`, `/:collection/:id`). |
| **Parameters** | URL query string parameters (e.g. `?fields`, `?sort`, `?q`). |
| **Metadata** | HTTP headers (e.g. `X-CRUD-Version`, `X-Correlation-ID`, `Prefer`). |
| **Content negotiation** | `Accept` request header and `Content-Type` response header. |
| **Error signaling** | HTTP status codes (`4xx`, `5xx`) with an error body payload. |

### How to read endpoint rules

Rules are organized by endpoint signature. Each endpoint has two tabs:

* **Specific rules** — constraints that apply only to this endpoint or a subset of endpoints.
* **Inherited rules** — global HTTP rules that apply to all endpoints.

Each rule reads as a phrase: *subject* `REQUIREMENT` *predicate*. Rules tagged with base spec codes (e.g. `SUB_ID`) indicate which [Base Specification](/0.1/base/) rule they implement — hover for the full text.

## 6. Asyncability

***

This section maps the [Asyncability extension](/0.1/extensions/async/) to HTTP. It defines how Clients and Servers negotiate synchronous vs. asynchronous operation execution using HTTP headers and status codes. These rules apply globally to all endpoints.

### 6.1 Client negotiation

<SectionDescription sectionCode="HA61" />

| Client ability | HTTP Header |
|---|---|
| **Sync-only** (default) | No async-specific header. MAY send `Expect: 200-ok`. |
| **Async-preferred** | `Prefer: respond-async` |
| **Async-required** | `Expect: 202-accepted` |

<RulesCardGrid sectionCode="HA61" open />

### 6.2 Server response

<SectionDescription sectionCode="HA62" />

| Server × Client | Sync-only Client | Async-preferred Client | Async-required Client |
|---|---|---|---|
| **Sync-only Server** | `200` | `200` (MAY `417`) | MUST `417` |
| **Async-only Server** | MUST `417` | `202` | `202` |
| **Both Server** | `200` | `202` | `202` |

<RulesCardGrid sectionCode="HA62" open />

### 6.3 Operation polling

<SectionDescription sectionCode="HA63" />

| Polling result | HTTP Response |
|---|---|
| Still running | `425 Too Early` |
| Completed | `200 OK` with operation results |
| Failed | `500 Internal Server Error` with error payload |

## 7. Resource Metadata

***

This section maps the [Metadata extension](/0.1/extensions/metadata/) to HTTP. Resource-level Metadata is exposed through standard HTTP headers alongside the response body.

### 7.1 Versioning headers

<SectionDescription sectionCode="HM71" />

| Header | Direction | Purpose |
|---|---|---|
| `ETag` | Response | Document hash for change detection. |
| `If-Match` | Request | Send current ETag on writes for optimistic concurrency. Server returns `412` on mismatch. |
| `If-None-Match` | Request | Send current ETag on reads for conditional caching. Server returns `304` if unchanged. |
