This Protocol Specification maps the [CRUD API Base Specification](/0.1/base/) to the **Model Context Protocol (MCP)**, as defined in [Protocols](/0.1/protocols/). It follows the [MCP Specification (2025-03-26)](https://modelcontextprotocol.io/specification/2025-03-26) and uses [JSON-RPC 2.0](https://www.jsonrpc.org/specification) as the message format.

The five protocol concepts are realized as follows:

| Concept | MCP realization |
|---|---|
| **Operations** | JSON-RPC methods with `crud/` namespace (e.g., `crud/get`, `crud/list`). |
| **Parameters** | Top-level fields in the JSON-RPC `params` object (e.g., `fields`, `sort`, `query`). |
| **Metadata** | `_meta` object within `params` (request) and `result` (response), following the MCP `_meta` convention. |
| **Content negotiation** | `experimental.crud` capability declared during the MCP `initialize` handshake. |
| **Error signaling** | JSON-RPC error object with CRUD-specific codes in the `-32000` to `-32099` range. |

## 1. Methods

***

<SectionDescription sectionCode="M1" />

| CRUD Operation | MCP Method | Subject in `params` |
|---|---|---|
| `get` | `crud/get` | `collection` + `id` |
| `list` | `crud/list` | `collection` + optional `query` or `view` |
| `create` | `crud/create` | `collection` + `documents` |
| `patchById` | `crud/patch` | `collection` + `id` + `patch` |
| `patchByQuery` | `crud/patch` | `collection` + `query` + `patch` |
| `replaceById` | `crud/replace` | `collection` + `id` + `document` |
| `replaceByQuery` | `crud/replace` | `collection` + `query` + `document` |
| `deleteById` | `crud/delete` | `collection` + `id` |
| `deleteByQuery` | `crud/delete` | `collection` + `query` |

## 2. Parameters

***

<SectionDescription sectionCode="M2" />

All parameters are top-level fields in `params`, alongside Subject fields:

| Parameter | Field | Type | Operations |
|---|---|---|---|
| Projection | `fields` | `string` | All except `crud/delete` |
| Sorting | `sort` | `string` | `crud/list` |
| Pagination | `page`, `limit` | `number` | `crud/list` |
| Expect match | `expectMatch` | `number` | `crud/patch`, `crud/replace`, `crud/delete` (batch) |
| Deny upsert | `denyUpsert` | `boolean` | `crud/replace` (by id) |

<RulesCardGrid sectionCode="M2" open />

## 3. Metadata

***

<SectionDescription sectionCode="M3" />

The `_meta` object in requests:

| Field | Type | Requirement | Description |
|---|---|---|---|
| `crudVersion` | `string` | MUST | CRUD API Spec version |
| `crudExtensions` | `string[]` | MUST | Supported extension names |
| `correlationId` | `string` | SHOULD | Fresh `uuid-hex` |
| `prefer` | `string` | MAY | `"representation"` (default) or `"status"` |
| `handling` | `string` | MAY | `"lenient"` (default) or `"strict"` |

The `_meta` object in responses:

| Field | Type | Requirement | Description |
|---|---|---|---|
| `crudVersion` | `string` | MUST | Server's CRUD API Spec version |
| `correlationId` | `string` | MUST | Echoed or newly generated |
| `matchedCount` | `number` | SHOULD | For query-based operations |
| `modifiedCount` | `number` | SHOULD | For write operations |
| `upserted` | `boolean` | MUST | When replace creates a new Document |
| `warnings` | `string[]` | MAY | Validation warnings |

<RulesCardGrid sectionCode="M3" open />

***

<SectionDescription sectionCode="M4" />

|---|---|
| `crud/get` | Document fields at top level + `_meta` |
| `crud/list` | `{ documents: [...], _meta: {...} }` |
| `crud/create`, `crud/patch`, `crud/replace` | `{ documents: [...], _meta: {...} }` |
| `crud/delete` | `{ _meta: {...} }` |
| Any (with `prefer: "status"`) | `{ _meta: {...} }` |

## 5. Error handling

***

<SectionDescription sectionCode="M5" />

| Error | Code | Equivalent HTTP |
|---|---|---|
| Not Found | `-32001` | `404` |
| Conflict | `-32002` | `409` |
| Precondition Failed | `-32003` | `412` |
| Negotiation Failed | `-32004` | `400` / `406` |

These codes are in the JSON-RPC Server-defined error range (`-32000` to `-32099`). Standard JSON-RPC errors (`-32600` to `-32603`) remain unchanged.

## 6. Capability negotiation

***

<SectionDescription sectionCode="M6" />

During the MCP `initialize` handshake, the Server declares CRUD support via the `experimental.crud` capability object. This replaces per-request version and extension negotiation — Clients that have completed initialization already know the Server's CRUD version and capabilities.

## 7. Asyncability

***

<SectionDescription sectionCode="MA7" />

Unlike HTTP (which uses `Prefer`/`Expect` headers and status codes), MCP leverages the `_meta` object for negotiation and JSON-RPC's native notification system for progress tracking.

### 7.1 Negotiation

<SectionDescription sectionCode="MA71" />

| Client ability | `_meta.async` value |
|---|---|
| **Sync-only** (default) | Field absent |
| **Async-preferred** | `"prefer"` |
| **Async-required** | `"require"` |

<RulesCardGrid sectionCode="MA71" open />

### 7.2 Server response

<SectionDescription sectionCode="MA72" />

| Server × Client | Sync-only Client | Async-preferred | Async-required |
|---|---|---|---|
| **Sync-only Server** | Normal result | Normal result (MAY `-32005`) | MUST `-32005` |
| **Async-only Server** | MUST `-32005` | Async accepted | Async accepted |
| **Both Server** | Normal result | Async accepted | Async accepted |

When accepted, the response contains `_meta.async: "accepted"` and `_meta.operationId`.

<RulesCardGrid sectionCode="MA72" open />

### 7.3 Operation polling

<SectionDescription sectionCode="MA73" />

| `crud/operationResult` response | `_meta.async` |
|---|---|
| Still running | `"pending"` |
| Completed | Field absent (full result returned) |
| Failed | JSON-RPC error |

<RulesCardGrid sectionCode="MA73" open />

### 7.4 Progress notifications

<SectionDescription sectionCode="MA74" />

The Server uses the standard MCP `notifications/progress` mechanism with the `progressToken` provided by the Client. This gives real-time feedback without polling.

<RulesCardGrid sectionCode="MA74" open />

### 7.5 Callback notifications

<SectionDescription sectionCode="MA75" />

Instead of webhook URLs (as in HTTP), MCP uses JSON-RPC notification methods. The Client specifies method names that the Server invokes as notifications.

<RulesCardGrid sectionCode="MA75" open />

## 8. Resource Metadata

***

This section maps the [Metadata extension](/0.1/extensions/metadata/) to MCP. Resource-level Metadata is nested inside the `_meta` object that is already part of every CRUD result.

### 8.1 Resource Metadata in `_meta`

<SectionDescription sectionCode="MM81" />

| `_meta` field | Type | Requirement | Maps to |
|---|---|---|---|
| `version` | `number` | MUST | Document version counter |
| `hash` | `string` | SHOULD | Document hash (CRC32 base64) |
| `events.created` | `object` | MUST | Creation event (timestamp, author) |
| `events.updated` | `object` | MUST (on writes) | Last write event |
| `events.published` | `object` | MAY | Publication event |
| `events.archived` | `object` | MAY | Archival event |
| `status` | `string` | MAY | Lifecycle: `"draft"` / `"published"` / `"archived"` |
| `ancestor` | `string` | MUST (if cloned) | Original Document Resource ID |
| `origin.realm` | `string` | MAY | External source hostname/workspace |
| `origin.type` | `string` | MAY | External source Resource Type |

For optimistic concurrency, the Client sends `_meta.ifMatch` in write requests with the expected hash. If mismatched, the Server returns error `-32003` (Precondition Failed).
