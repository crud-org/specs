This extension is OPTIONAL. CRUD APIs that do not need relationship tracking are not required to implement any of the rules in this section.

The `$rel` convention follows the `$`-prefix reserved for metadata about Properties, as defined in [Section 3.1.2 Resource Fields](/0.1/base/general/#312-resource-fields). Unlike regular metadata that Servers may strip, `$rel` fields are structural — they define the shape of the Document and MUST be preserved.

## 1. Relationship structure

***

<SectionDescription sectionCode="R11" />

Each relationship Property is a JSON object with:

| Field | Type | Required | Description |
|---|---|---|---|
| `$rel` | `string` | MUST | Target Collection slug or external URI. |
| `value` | `string \| string[] \| null` | MUST | Resource ID(s) of the referenced Document(s). |
| *(other)* | any | MAY | Denormalized fields for convenience (non-normative). |

## 2. Relationship targets

***

<SectionDescription sectionCode="R12" />

| Target type | `$rel` format | Example |
|---|---|---|
| **Internal** | Collection slug | `"users"`, `"tags"` |
| **External** | URI or URL | `"google.com/categories"`, `"https://api.example.com/products"` |

External relationships are opaque references — the Server stores them but does not resolve or validate them.

<RulesCardGrid sectionCode="R12" open />

## 3. Cardinality

***

<SectionDescription sectionCode="R13" />

| Cardinality | `value` type | Empty state |
|---|---|---|
| **Single** (one-to-one) | `string` (Resource ID) | `null` |
| **Multiple** (one-to-many) | `string[]` (Resource ID array) | `[]` |
